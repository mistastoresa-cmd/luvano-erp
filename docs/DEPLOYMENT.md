# دليل النشر — لوفانو ERP (Neon + Vercel)

بيئة الإنتاج تستخدم **Neon** (Postgres) و**Vercel** (استضافة Next.js). الكود
جاهز لهذا أصلاً: `db/client.ts::createDb()` يتحوّل تلقائياً لسائق Neon متى ما
كان `DATABASE_URL` موجوداً — بدون أي تغيير في الكود. طبقة pglite المحلية
(`.data/`) لا تُستخدم إطلاقاً في الإنتاج.

> ⚠️ **الأسرار تدخلها أنت بحسابك.** لا تشارك `DATABASE_URL` ولا
> `BETTER_AUTH_SECRET` في أي مكان عام (شات، commit، سكرين شوت).

---

## 1. قاعدة البيانات — Neon

1. أنشئ حساباً على [neon.tech](https://neon.tech) ومشروعاً جديداً (اختر أقرب
   منطقة — مثلاً Frankfurt لأداء أفضل من السعودية).
2. من صفحة المشروع، انسخ **Pooled connection string** (يحتوي `-pooler`
   و`?sslmode=require`). هذا هو `DATABASE_URL`.

## 2. مفتاح المصادقة — Better Auth

ولّد سرّاً عشوائياً مرة واحدة:

```bash
npx @better-auth/cli secret
# أو: openssl rand -base64 32
```

احتفظ بالناتج — هذا `BETTER_AUTH_SECRET`.

## 3. تطبيق الهجرات على Neon

من جهازك، شغّل الهجرات مرة واحدة ضد قاعدة Neon (استبدل السلسلة بسلسلتك):

```bash
DATABASE_URL='postgres://…-pooler…?sslmode=require' npm run db:migrate
```

يطبّق ملفات `db/migrations/0000…0009` على القاعدة الفارغة. أعد تشغيله بعد أي
`npm run db:generate` مستقبلاً.

## 4. النشر على Vercel

1. ارفع المستودع على GitHub (خاص).
2. على [vercel.com](https://vercel.com): **Add New → Project** → استورد
   المستودع. Vercel يكتشف Next.js تلقائياً.
3. في **Environment Variables** أضف (لبيئتَي Production وPreview):

   | المتغيّر | القيمة |
   |---|---|
   | `DATABASE_URL` | سلسلة Neon المجمّعة (pooled) |
   | `BETTER_AUTH_SECRET` | السرّ من الخطوة 2 |
   | `BETTER_AUTH_URL` | رابط النشر، مثلاً `https://luvano.vercel.app` |

   > `BETTER_AUTH_URL` **لازم** يطابق الدومين الفعلي، وإلا تنكسر الكوكيز
   > وإعادة التوجيه. حدّثه لو ربطت دومين مخصّص لاحقاً.
4. **Deploy**. البناء (`next build`) يمرّ نظيفاً (مؤكَّد محلياً).

## 5. إنشاء أول مستأجر (شركة + مالك)

لا تستخدم `db:seed` في الإنتاج (بيانات تجريبية). أنشئ مستأجرك الحقيقي عبر
مسار الإعداد — بعد النشر شغّل (استبدل القيم):

```bash
curl -X POST https://<دومينك>/api/onboarding \
  -H 'Content-Type: application/json' \
  -d '{
    "companyName": "متجر ميستا للعطور",
    "ownerName": "عبدالله الباقي",
    "ownerEmail": "you@example.com",
    "ownerPassword": "كلمة-مرور-قوية"
  }'
```

ثم افتح `https://<دومينك>/login` وسجّل الدخول بنفس البيانات.

> ملاحظة: شجرة الحسابات وربط الحسابات المحاسبية (`account_mappings`) لا
> يُنشئها مسار الإعداد بعد — أول فاتورة تحتاج ترحيل محاسبي ستحتاج إعداد هذه
> الحسابات (حالياً تُنشأ فقط في `scripts/seed.ts`). عند بناء شاشة إعداد
> الحسابات في موديول المحاسبة، ستُغطّى هذه الفجوة.

## ملاحظات مهمة

- **الهجرات ليست جزءاً من بناء Vercel** عمداً (Vercel لا ينبغي أن يشغّل DDL
  تلقائياً على كل نشر). طبّقها يدوياً بالخطوة 3 عند إضافة جداول جديدة.
- **`serverExternalPackages: ['@electric-sql/pglite']`** في
  `next.config.ts` غير ضار في الإنتاج — pglite ببساطة لا يُستدعى لأن
  `DATABASE_URL` موجود.
- **الاختبارات** (`npm test`) تظل تستخدم pglite في الذاكرة — لا تحتاج Neon.
- **جلسات قصيرة** (30 دقيقة، `lib/auth/server.ts`) — سلوك مقصود من مراجعة
  RBAC؛ المستخدم يبقى مسجّلاً طالما يستخدم النظام (تجديد كل 5 دقائق).

## ما ليس جاهزاً بعد للإطلاق الكامل

الواجهة الحالية تغطّي **المبيعات + المخزون + الدورة المستندية** فقط. الموديولات
التالية لها backend مختبَر لكن بلا شاشات (تظهر "قريباً"): المشتريات، المحاسبة،
العملاء، التسويق، الموارد البشرية، والتقارير المفصّلة. مناسبة كـ**staging**
للتجربة والبناء الآن؛ الإطلاق الكامل للعميل الأول يحتاج إكمال الشاشات الأساسية.
