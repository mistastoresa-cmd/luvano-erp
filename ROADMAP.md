## 1. نموذج العمل (ملخّص من وثيقة التصميم المعتمدة)

- **المشكلة المُثبَتة بدليل:** شركات SMB بفروع فعلية + متجر سلة تعاني من فجوة مزامنة
  مخزون حقيقية — كميات وهمية أو تعديل يدوي متكرر (`docs/ARCHITECTURE.md` → الوثيقة
  الأصلية).
- **الشريحة المستهدفة:** مالك/مدير أو محاسب SMB، مسعّرة برة إمكانياتها من Odoo.
- **الميزة التنافسية:** ربط أصلي (native) مع سلة — لا Odoo ولا الأنظمة المحلية عندها هذا اليوم.
- **نطاق سعري مبدئي (لم يُختبر بعد):** 300-800 ريال/شهر لكل فرع إضافي فوق الأول.
  **⚠️ لسه ما تحقق مع عميل خارجي حقيقي — انظر البند د.1 أدناه.**
- **قناة التوزيع:** SaaS مباشر أولاً؛ سلة App Store قناة مستقبلية محتملة (غير مصمَّمة).

## 2. معايير "تم" لكل بُعد

### أ. الاكتمال الوظيفي (منطق تطبيق حقيقي، مو schema فقط)

| القسم | معيار "تم" |
|---|---|
| المخزون + الفوترة | ⚠️ الحركات/الفوترة منجزة (ذرية، idempotent). **فجوة مكتشفة 2026-07-21:** بلا مفهوم منتج/متغيرات — `sku` نص مسطّح بـ8 جداول. **تم:** `products`+`product_variants` (أب/ابن، ألوان/مقاسات)، تقارير/عروض تستهدف الأب (كل الأبناء) أو ابن محدد |
| المستودعات + التحويلات | ✅ منجز — نقل كمية+تكلفة بين فروع |
| المشتريات | ⚠️ جزئي — استلام فقط. **تم:** ترحيل أمر شراء تلقائي (draft→sent→received) |
| المحاسبة | ✅ منجز — قيود متوازنة، COGS، حالة دفع تلقائية |
| التقارير | ⚠️ جزئي (لكل فرع). **تم:** تقرير شركة موحّد (تجميع كل الفروع) |
| العملاء | ❌ schema فقط. **تم:** CRUD حقيقي + سجل تعاملات |
| التسويق والعروض | ❌ schema فقط. **تم:** تفعيل/تعطيل كوبون + حساب استخدام فعلي |
| الموارد البشرية | ❌ schema فقط. **تم:** حساب راتب فعلي + ترحيل محاسبي (`payroll_entries.journalEntryId` محجوز أصلاً) |

### ج. واجهة مستخدم فعلية

غير مصمَّمة بعد بالتفصيل. **الحد الأدنى المطلوب لأول إطلاق (MVP UI):**
- تسجيل دخول/دعوة أعضاء (يعتمد على RBAC)
- ربط متجر سلة (يعتمد على مبادرة ربط سلة)
- شاشة مخزون لكل فرع (قراءة أرصدة + حركات)
- شاشة فواتير بيع (قراءة + إدخال يدوي لفرع)
- شاشة تقارير أرباح/خسائر وميزانية لكل فرع (البيانات جاهزة، `lib/reporting/service.ts`)
- شاشة استلام مشتريات + تحويل مخزون

### د. جاهزية العمل

1. **التحقق الخارجي من الطلب** — تكليف وثيقة التصميم الأصلية (3 محادثات عملاء حقيقيين) لسه قائم كنشاط مستقل، مو مانع تقني للبناء.
2. **نموذج التسعير النهائي** — يحتاج تأكيد بعد محادثات العملاء، مو افتراض دائم.
3. **Onboarding** — تدفّق "شركة جديدة تسجّل → تنشئ فروعها → تربط سلة → تبدأ" غير مصمَّم كتجربة مستخدم متكاملة بعد (الأجزاء التقنية موجودة كخطط منفصلة).
4. **الدعم الفني** — غير مطروح بعد.

## 3. الترتيب التنفيذي — المراحل بالتبعيات

```
Phase 0 (منجز) ──▶ Phase 1 ──▶ Phase 1.5 ──▶ Phase 2 ──▶ Phase 3 ──▶ Phase 4 ──▶ Phase 5 ──▶ Phase 6
الأساس المحاسبي-   RBAC        إكمال الموديولات  ربط سلة    UI أساسي  جاهزية      جاهزية      جاهزية
المخزوني (5 خدمات) (T1-T6✅،    + المتغيرات       (مؤجَّل)   (MVP)     إنتاج       عمل         (بالتوازي
                    T7/T8 متبقٍ) (schema-فقط →                                                 من البداية)
                                  منطق حقيقي)
```

**تحديث 2026-07-21 (بطلب صريح):** ترتيب المراحل تغيّر — ربط سلة الحي (كان Phase 2) **مؤجَّل** حتى تكتمل باقي الموديولات والأقسام أولاً. السبب المعلن: التأكد من جاهزية النظام كموديولات كاملة قبل ربط أي مصدر بيانات حي عليه.

- **Phase 0 (منجز):** الوedge + المحاسبة + المستودعات + التقارير الفرعية.
- **Phase 1 — RBAC:** ✅ T1-T6 منفَّذة ومختبرة. متبقٍ T7 (ربط الخدمات الخمس بالتحقق الفعلي) وT8 (سجل تدقيق).
- **Phase 1.5 — إكمال باقي الموديولات (الأولوية الحالية):** العملاء، التسويق والعروض، الموارد البشرية (schema فقط → منطق حقيقي) + **متغيرات المنتج الجديدة** (أب/ابن — ألوان ومقاسات، انظر التفاصيل أدناه) + إكمال المشتريات (دورة أمر الشراء كاملة) + تقرير الشركة الموحّد (تجميع كل الفروع).
- **Phase 2 — ربط سلة الحي:** خطة مقفولة سابقًا (5 مهام T1-T5)، **مؤجَّلة** لحين اكتمال Phase 1.5. الخطة نفسها تبقى صالحة، بس التنفيذ ينتظر.
- **Phase 3 — UI أساسي:** يعتمد على RBAC + بيانات حقيقية من الموديولات المكتملة.
- **Phase 5 — جاهزية إنتاج:** بعد وجود ترافيك حقيقي فعلي (بعد Phase 2) لتصبح المراقبة/الاختبار ذات معنى.
- **Phase 6 — جاهزية عمل:** بالتوازي مع كل المراحل — التحقق من العملاء (البند د.1) يبدأ من الآن، مو ينتظر اكتمال البناء.

**القاعدة المتفق عليها من الآن فصاعدًا:** أي عمل تقني جديد يُخطَّط أولاً (`/plan-eng-review` أو ما يعادله) قبل التنفيذ، ويُربط بالمرحلة المناسبة أعلاه — لا بناء مستقل خارج هذا التسلسل.

## 4. الحالة التفصيلية للمبادرات المخطَّطة

- **RBAC (Phase 1):** ✅ T1-T6 منفَّذة ومختبرة (55 اختبار). Better Auth، تقييد فرع من أول يوم، حماية مزدوجة (proxy+خدمة)، 4 أدوار، جلسات قصيرة+إبطال فوري، تينانت من لوفانو أولاً عبر `lib/auth/provisioning.ts`. متبقٍ: T7 (ربط الخدمات الخمس فعليًا بـ`assertRole`/`assertBranchAccess`) وT8 (سجل تدقيق). 6 أخطاء حقيقية اتصلحت أثناء التنفيذ — تفاصيل كاملة في `docs/ARCHITECTURE.md`.
- **ربط سلة الحي (Phase 2):** خطة كاملة، 5 مهام (T1-T5)، صفر تنفيذ (عدا تصحيح خطأ حقيقي بـ`SallaConnector` دخل الإنتاج فعلاً). القرارات: قراءة فقط الآن، `salla_connections` مشفّرة، دايمًا 200 + dead-letter، فرع أونلاين محدد، تينانت من لوفانو أولاً، متجر تجريبي قبل عميل حقيقي.
